## はじめに

こんにちは。イオンスマートテクノロジー株式会社（AST）でSREチームの林 aka [もりはや](https://twitter.com/morihaya55)です。

本記事ではAkamaiのProperty ManagerにおけるCriteriaの[Time Interval](https://techdocs.akamai.com/property-mgr/docs/time-interval)を利用し、メンテナンス画面への切り替えを「指定時刻ちょうど」に開始する方法を紹介します。

当社ではCDN/WAFの機能に[Akamai社](https://www.akamai.com/ja)のサービスを採用しています。Akamaiは非常に強力で頼もしい一方、Propertyの設定変更を本番環境に反映するにはアクティベーションの時間が必要です。

通常の設定変更であれば数分程度の反映待ちは大きな問題になりません。しかし、計画メンテナンスの開始時刻が決まっている場合、「23:00ちょうどにメンテナンス画面を表示したい」のような要件が出てきます。

このときに「23:00になったらAkamaiの設定を反映（Activate）する」方式では、どうしても反映までのタイムラグが避けられませんでした。

そこで今回は、設定変更の反映自体は事前に完了させておき、指定時刻になった瞬間にAkamai CDN側でメンテナンス条件が成立するようにしました。

## TL;DR

本記事を箇条書きでまとめると以下です。

- 従来のAkamaiを利用したメンテナンス開始処理は、コンフィグ反映まで数分以上のタイムラグがあった
- Property ManagerのCriteriaには`Time Interval`があり、指定期間だけRuleを有効にできる
- `timezone=Asia/Tokyo`を明示することで、日本時間の指定時刻にメンテナンスを開始できた
- 終了時刻の有無、キャッシュ、Ruleの順序、Stagingでの検証は事前にしっかり確認する

## 背景

### メンテナンス開始時刻をできるだけ正確にしたい

Webサービスの計画メンテナンスでは、事前告知や関連システムとの調整により、開始時刻と終了時刻が決まっているケースがあります。

たとえば「23:00から05:00までHOGEサービスをメンテナンス表示にする」と決めた場合、利用者向けの告知や社内外の運用もその時間を基準に動きます。（HOGEサービスはサンプル用の仮名であり当社の実サービスではございません）

もちろん現実の運用では数秒単位まで厳密に保証できるものではありませんが、少なくとも「23:00開始と告知しているのに、Akamaiの反映待ちで23:15頃にようやく切り替わる」状況は避けたいところです...

### 設定反映を開始トリガーにするとタイムラグが出る

AkamaiのProperty Managerで設定を変更した場合、本番環境へ反映するためにはActivateが必要です。

このActivateは世界中のAkamai Edgeへ設定を配布する処理であり、即時反映ではありません。数分程度で終わることもありますが、タイミングや環境によってはもう少し時間がかかることもあります。

そのため、メンテナンス開始時刻に合わせて人間がControl Center（Web画面）からActivateを押す、あるいはAPIでActivateする方式では、どうしても以下のような課題が残ります。

- 反映完了までの時間を正確に読めない
- 開始時刻直前の手作業が発生する（これはAPIやAkamai CLIで解決済みでしたが）
- 反映が遅れた場合、告知した開始時刻と実際の挙動がズレる
- 切り戻しや終了時刻も同じ問題を抱える

そこで今回使ったのが、AkamaiのCriteriaにある`Time Interval`です。

https://techdocs.akamai.com/property-mgr/docs/time-interval

## AkamaiのCriteriaで時間を条件にする

### RuleはBehaviorとCriteriaで構成される

Akamai Property ManagerのRuleは、大きく分けると以下の要素で構成されています。

| 要素 | 役割 |
| --- | --- |
| Behavior | 条件に一致したときに何をするか |
| Criteria | どのリクエストに対してBehaviorを適用するか |

今回のメンテナンスでは「`/hogeservice`配下へのリクエスト」かつ「指定した時間帯のリクエスト」だけメンテナンス画面を返す条件になります。

そのため、Criteriaには以下の2つを設定しました。

- `Time Interval`: `2026/04/26 23:00:00`から`2026/04/27 05:00:00`の間に一致する ＊日付はサンプルです
- `Path`: `/hogeservice`と`/hogeservice/*`に一致する

![akamai-property](/images/morihaya-20260427-akamai-maintenance-time/2026-04-27-00-32-57.png)

### JSONでみる具体的な設定内容

Akamai CDNのProperty Managerには、設定状態をXMLまたはJSONで表示する機能があります。今日のAI Agentの隆興を鑑み本記事では具体的な設定をJSON形式で提示します。

まずはメンテナンスに関連する設定全体の紹介です。メンテナンスの発動のために通常のHOGEサービスのキャッシュRuleの後ろに`Hoge Maintenance`Ruleを追加しています。

実際の差分として重要なのは以下の部分です。

```json
{
  "name": "Hoge Maintenance",
  "behaviors": [
    {
      "name": "rewriteUrl",
      "options": {
        "behavior": "REWRITE",
        "targetUrl": "/maintenance/50x.html"
      }
    },
    {
      "name": "responseCode",
      "options": {
        "statusCode": 503
      }
    },
    {
      "name": "downstreamCache",
      "options": {
        "behavior": "BUST"
      }
    }
  ],
  "criteria": [
    {
      "name": "path",
      "options": {
        "matchOperator": "MATCHES_ONE_OF",
        "values": [
          "/hogeservice",
          "/hogeservice/*"
        ]
      }
    },
    {
      "name": "time",
      "options": {
        "matchOperator": "BETWEEN",
        "beginDate": "2026/04/26 23:00:00 timezone=Asia/Tokyo",
        "endDate": "2026/04/27 05:00:00 timezone=Asia/Tokyo"
      }
    }
  ],
  "criteriaMustSatisfy": "all"
}
```

:::message
上記は記事用に重要部分だけを抜粋した例です。実際のRuleにはメンテナンスHTMLを返すためのOrigin設定なども含まれています。
:::

## 実装内容

### `/hogeservice`配下だけを対象にする

まず、メンテナンス対象を`/hogeservice`配下に限定しています。

```json
{
  "name": "path",
  "options": {
    "matchOperator": "MATCHES_ONE_OF",
    "matchCaseSensitive": false,
    "normalize": false,
    "values": [
      "/hogeservice",
      "/hogeservice/*"
    ]
  }
}
```

この条件により、トップページや他のパスには影響を与えず、HOGEサービスだけをメンテナンス対象にできます。

メンテナンス対応は影響範囲を小さくすることが大切です。全体メンテナンスが必要なケースもありますが、対象パスを絞れるなら絞った方が、検証も切り戻しも楽になります。

### `time` Criteriaで開始時刻と終了時刻を指定する

今回の主役がこの`time` Criteriaです。
Web画面には`Time Interval`と表示されますが、JSONコンフィグ上ではシンプルに`time`として扱われています。

```json
{
  "name": "time",
  "options": {
    "matchOperator": "BETWEEN",
    "beginDate": "2026/04/26 23:00:00 timezone=Asia/Tokyo",
    "endDate": "2026/04/27 05:00:00 timezone=Asia/Tokyo"
  }
}
```

ポイントは`timezone=Asia/Tokyo`を明示していることです。

タイムゾーンを曖昧にしたまま時間指定を扱うと、UTCなのかJSTなのかで読み違いが発生します。とくにメンテナンスのように時刻そのものが重要な作業では、設定値を見た人が迷わない形にしておくことが大切です。GUIからはわかりやすく指定必須となっているためデフォルトのGMTからAsia/Tokyoへ変更しましょう。

また、`BETWEEN`で開始時刻と終了時刻をまとめて指定できるため、メンテナンス解除のために終了時刻ぴったりに再度Activateする必要もありません。

設定は事前に本番反映しておき、開始時刻になると条件が成立してメンテナンス表示になり、終了時刻を過ぎると条件が成立しなくなって通常のRuleへ戻る流れです。

### メンテナンスHTMLを返す

メンテナンス中は、通常のOriginへリクエストを流すのではなく、Akamaiから別のOriginへ向ける形でメンテナンスHTMLを返す構成にしました。

設定では`rewriteUrl`を使い、対象リクエストを`/maintenance/50x.html`へ向けています。

```json
{
  "name": "rewriteUrl",
  "options": {
    "behavior": "REWRITE",
    "targetUrl": "/maintenance/50x.html"
  }
}
```

これにより、アプリケーション側やHOGEサービス本体のOriginに依存せず、Akamai側でメンテナンス表示を完結できます。

### HTTP 503を返す

メンテナンス表示では、HTTPステータスコードとして`503 Service Unavailable`を返すようにしました。

```json
{
  "name": "responseCode",
  "options": {
    "statusCode": 503
  }
}
```

ユーザーのブラウザにメンテナンスページが表示されるだけであれば、HTTP 200でも見た目上は成立してしまいます。

しかし、計画メンテナンスで一時的にサービスを提供できない状態であることを機械的にも伝えるなら、503を返す方が自然です。

検索エンジンや外部監視、社内の監視基盤に対しても「ページは存在するが正常サービスではない」と伝えられます。

### downstreamCacheはBUSTにする

もうひとつ重要なのがキャッシュです。

```json
{
  "name": "downstreamCache",
  "options": {
    "behavior": "BUST"
  }
}
```

メンテナンスページがブラウザや途中のキャッシュに残ってしまうと、終了時刻を過ぎても利用者の手元でメンテナンス画面が見え続ける可能性があります。

そのため、メンテナンス表示では下流キャッシュに残さない設定を入れています。

CDNの設定では「切り替わること」だけでなく「戻ること」も同じくらい大切です。とくにメンテナンス終了後は、利用者に通常画面が戻っていることをすぐ確認してもらいたいため、キャッシュの扱いは強く意識しました。

## この方式の良いところ

### 事前にActivateを完了できる

一番大きな利点は、メンテナンス開始前にAkamaiのActivateを完了できることです。

開始時刻の数十分前、あるいは余裕を持ってそれ以上前に設定を反映しておけば、開始直前に人間がControl Center（Web画面）を操作する必要がありません。

| 方式 | 開始時刻の正確さ | 作業負荷 | 反映遅延の影響 |
| --- | --- | --- | --- |
| 開始時刻にActivateする | 読みにくい | 開始直前に作業が必要 | 受ける |
| `Time Interval` Criteriaを使う | 比較的合わせやすい | 事前反映で済む | 受けにくい |

人間の操作を開始時刻から切り離せるのは、運用上かなり大きな安心材料です。

### 終了も自動化できるし、手動で解除もできる

`beginDate`だけでなく`endDate`も指定しているため、終了時刻を過ぎるとメンテナンスRuleの条件が成立しなくなります。

つまり、終了時刻に合わせて通常状態へ戻すためのActivateも不要です。

もちろん、作業完了確認やサービス正常性の確認は別途必要です。しかし、CDN側のメンテナンス表示のON/OFFを時間条件に任せられるだけでも、作業当日の緊張感はかなり下がります。

また、サービスによってはメンテナンスをできるだけ短く終わらせたいケースもあるでしょう。その場合は単純に`Time Interval`を追加する前のバージョンに戻すことで、事前に設定した`endDate`の時間よりも早くメンテナンスを解除できます。ただし、この場合は反映までラグが発生します。

### アプリケーションに手を入れずに実現できる

今回の方式はAkamai側で完結しています。

アプリケーション側にメンテナンスモードを実装したり、Origin側でリダイレクトや503応答を組み込んだりする必要がありません。

もちろんアプリケーション側のメンテナンス制御が適しているケースもあります。ただ、CDNの入口で対象パスをまとめて制御したい場合には、Akamai側で実装できることは大きな選択肢になります。

## 注意点

### Stagingで必ず確認する

時間指定の設定は見た目にはシンプルですが、実際の挙動確認が重要です。

とくに以下はAkamai Stagingで確認しておくべきでしょう。ミスって本番を予定時間外にメンテナンスにすると目も当てられません。

- 指定時刻前は通常ページが返るか
- 指定時刻内はメンテナンスページが返るか
- HTTPステータスコードが503になっているか
- 指定時刻後は通常ページへ戻るか
- 対象外パスに影響がないか
- PC/スマートフォンなどデバイス別の既存Ruleに影響がないか

メンテナンス対象の他に既存のキャッシュRuleやデバイス判定などがある場合、Ruleの順序やCriteriaの組み合わせによって期待と異なるRuleが効く可能性があります。

「設定としては正しそう」ではなく、「Akamai Edgeとして期待通りに判定される」ことをしっかりとStagingで確認しましょう。

### Ruleの順序に注意する

AkamaiのRule Treeでは、どのRuleの配下に置くか、同じ階層のどの位置に置くかが重要です。

今回の`Hoge Maintenance`は、HOGEサービス関連のRule配下に追加しています。`path`と`time`のCriteriaを持っているとはいえ、既存Ruleとの関係を見誤ると、メンテナンスRuleが期待通りに効かなかったり、逆に広く効きすぎたりする可能性があります。

メンテナンスRuleは強い挙動を持つRuleです。HTTP 503を返す設定であるため、影響範囲の確認は普段のキャッシュ設定変更以上に慎重に行うべきです。

### 終了時刻をどう設計するか

`endDate`を入れることで自動的に通常状態へ戻せる一方、メンテナンス作業が延長した場合の扱いも考えておく必要があります。

たとえば05:00終了予定として設定していたものの、Origin側の作業が長引いた場合、05:00を過ぎるとAkamai側のメンテナンス表示は解除されます。

そのため、運用としては以下のいずれかを決めておくとよいです。

- 終了時刻を少し余裕を持って設定する
- 延長の可能性がある場合は、メンテナンス解除は手動とし、`endDate`は指定しない
- 予定より早く終わった場合に通常化する手順も決めておく

時間ピッタリに開始できることと、作業全体が予定通り終わることは別問題です。

Akamaiの`Time Interval` Criteriaは非常に便利ですが、運用設計まで含めて利用するのがよいでしょう。

## おわりに

以上が「Akamaiを利用したメンテナンスを時間ピッタリに開始する方法」の紹介でした。

今回のポイントは、所要時間の読めないAkamaiの設定反映（Activate）をメンテナンス開始トリガーにしないことです。

PropertyのActivateは事前に完了させておき、実際の切り替えはEdge側の`Time Interval` Criteriaで行う。この考え方にすると、Akamaiの反映待ちによるズレを気にせず、指定時刻にちょうどでメンテナンス表示を開始できます。
